## 剩余问题：async 恢复时序差异

### 根因

**dartic 的 `_registerAwaitCallbacks` 总是用 `scheduleMicrotask` 恢复帧，但原生 Dart 的 `_Future._complete()` 是同步传播的。**

Dart SDK 的 `_StreamIterator._onData()` 内部逻辑：

```dart
void _onData(T data) {
  ...
  _stateData = data;          // _hasValue = true
  moveNextFuture._complete(true);  // 原生 Dart: 同步传播 → await for body 立即执行
                                   // → body 内 moveNext() 设 _hasValue = false
  if (_hasValue) _subscription?.pause();  // 原生 Dart: _hasValue 已为 false → 不 pause
                                          // dartic: body 还没跑 → _hasValue 仍为 true → pause!
}
```

原生 Dart 中 `_complete(true)` 同步执行整个 await for body（包括下一次 `moveNext()` 调用），所以 `_hasValue` 在 `if` 检查时已为 `false`。dartic 中 `.then()` 回调只是 `scheduleMicrotask(resume)`，body 延迟到下一个微任务才执行，导致 `_hasValue` 仍为 `true`，触发 `subscription.pause()`。

### 影响（3 个 Asynchronous_For_in 测试）

| 测试 | 状态 | 表现 |
|------|------|------|
| execution_A02_t01 | ERROR/timeout | pause + controller 未 close → 死锁 |
| execution_A02_t02 | FAIL | `sc.isPaused` 意外为 `true`（因多余 pause） |
| execution_A05_t05 | ERROR/timeout | pause 打乱微任务时序，`sc.add(42)` 未及时传递 |

### 与 07-async.md 已知局限的关系

这正是 `docs/design/07-async.md` **"已知局限与演进路径"第 4 条**的直接后果：

> await 无同步快速路径 — 每次 await 均让出 microtask，已完成 Future 也不例外。
> Phase 2：检测已完成 Future，在分发循环内直接恢复（不让出），减少 microtask 切换。

### 修复方向

在 `_registerAwaitCallbacks` 的 `.then()` 回调中，检测 Future 是否已同步完成（`_Future._complete` 内部调用）：如果是，直接同步恢复帧，而非 `scheduleMicrotask`。这需要区分两种场景：

1. **Future 尚未完成**（正常 await）：仍用 `scheduleMicrotask` 恢复
2. **Future 在 `.then()` 注册时已完成或 `._complete()` 同步触发回调**：直接恢复帧

实现方式（参考 Dart SDK `_Future` 内部）：
- `_Future._complete()` 调用 `_propagateToListeners()` 时，如果 listener 的 callback 是同步调用的，可以通过一个标志位检测
- 或者：在 `.then()` 回调中设置 `frame.resumeValue` 后检查当前是否已在分发循环内——如果是，直接写入恢复值而不 scheduleMicrotask

**注意**：这个优化影响面广（所有 await 点），需要独立方案，不应作为 await for 的修复。

## 剩余问题 2：switch + await for + break/continue 跨 label

### 根因

**`switch` 语句内的 `await for` 配合 `break`/`continue` 跳到外层 label 时，inlined finally 中的 AWAIT 恢复后未正确继续执行。**

这是编译器层面的问题。CFE 将 `await for` 解糖为 `try { while (await moveNext()) ... } finally { await cancel() }`。当 `break L` 跨越这个 try-finally 边界时，编译器调用 `_inlineFinalizersFromDepth` 内联 finally 块（包含 `await cancel()`）。问题出在 switch 语句包裹 await for 时，inlined finally 的 AWAIT 恢复后跳转目标不正确。

### 证据

- t01-t09 全部通过（Break 34 pass + Continue 37 pass），**只有 t10 失败**
- t10 是 **唯一** 组合 `switch + await for + break/continue to outer label` 的测试（grep 确认 t01-t09 无 switch 语句）
- t08/t09 使用 StreamController + await for + break（无 switch）→ 通过
- Break/t10 表现：async 函数静默退出，未到达 ASYNC_RETURN
- Continue/t10 表现：cancel log 不完整（`[1]` instead of `[1, 4, 2]`），嵌套 stream cancel 跟踪丢失

### 影响（2 个 async_loops_t10 测试）

| 测试 | 状态 | 表现 |
|------|------|------|
| Break/async_loops_t10 | ERROR/timeout | async 函数未完成，`asyncEnd()` 未调用 |
| Continue/async_loops_t10 | ERROR/timeout | cancel log 不完整，断言失败 |

### 修复方向

排查 `_compileSwitchStatement`（compiler_statements.dart:215-308）与 `_inlineFinalizersFromDepth` 的交互。当 break/continue 从 switch case 内部跨越 await for 的 try-finally 时，inlined finally 中的 AWAIT 恢复后需要正确跳转到 break 目标而非 switch 的下一条指令。

**关键文件**：
- `compiler_statements.dart:348-380` — `_compileLabeledStatement`, `_compileBreakStatement`
- `compiler_statements.dart:215-308` — `_compileSwitchStatement`
- `compiler_statements.dart:517-565` — `_compileTryFinally`

## 核心发现：CFE 已做解糖

**CFE 的 `ForInLowering` 转换在 dartic 编译器之前执行**。`await for` 不会以 `ForInStatement(isAsync: true)` 形态到达 dartic 编译器，而是被解糖为：

```dart
// 输入:
await for (var x in stream) { body }

// CFE 解糖输出:
{
  _StreamIterator<T> :for-iterator = new _StreamIterator<T>(stream);
  try {
    while (await :for-iterator.moveNext()) {
      var x = :for-iterator.current;
      body
    }
  } finally {
    if (:for-iterator._subscription != null)
      await :for-iterator.cancel();
  }
}
```

**关键推论**：
1. **不需要 `AWAIT_STREAM_NEXT` 新操作码**——现有 `AWAIT` 指令已能处理 `await moveNext()` 的挂起/恢复
2. **不需要修改编译器**——解糖后的 `while + await + try-finally` 已被正确编译为现有字节码
3. **唯一阻塞点是 Stream Bridge**——`_StreamIterator(stream)` 的构造函数参数 `stream` 必须是真正的 `Stream` 实例
